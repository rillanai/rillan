# ADR-013: Optional Graphify Integration for Structural Retrieval

## Status

Proposed

## Context

Rillan already has a bounded retrieval layer (ADR-003, ADR-009) that uses embedded SQLite for chunk storage and reciprocal rank fusion for hybrid lexical/vector retrieval. That layer treats the repository as a bag of chunks keyed by path, content, and vector similarity — it does not reason about call graphs, import edges, or symbol-level relationships between files.

The repository separately maintains a local knowledge graph under `graphify-out/` produced by the `graphify` tool. Graphify extracts code structure (functions, imports, references) and semantic edges between code and documentation (notably ADR↔code links). It is built as a local artifact by post-commit and post-checkout git hooks, stays gitignored, and is explicitly *not* part of the repository's authoritative source of truth.

The structural and semantic information graphify captures is the same kind of information Rillan's retrieval would otherwise need to reconstruct at index time — and some of it (e.g. ADR↔code semantic edges) is genuinely hard to derive from chunk-level retrieval at all. Using graphify as an *optional* input to retrieval lets Rillan answer "show me everything that touches `ChatCompletionsHandler`" with neighborhood traversal rather than brittle keyword matching, and lets policy reason about which ADRs govern a given file without hand-maintained tags.

The tension is that graphify is a developer-local tool, not a required runtime dependency. Rillan is a local-first daemon with strict guardrails (ADR-002 bind locality, ADR-003 one-root manual indexing, ADR-012 audit/policy authority). Any integration must not:

- make graphify a hard dependency for request success,
- introduce background watchers, network calls, or external-service coupling,
- copy graphify artifacts into Rillan's authoritative storage,
- trust structure that graphify itself marked as unreliable.

The decision below frames how Rillan consumes graphify when it's present and how it degrades when it is not. This ADR does not commit to an implementation milestone; it ratifies the constraints a follow-on milestone must honour.

## Decision

Rillan will treat graphify output as an *optional, advisory* retrieval input under the following rules.

1. **Optionality is absolute.**
   - The authoritative retrieval path remains the chunk/vector store from ADR-003 and ADR-009.
   - Absence, corruption, or staleness of `graphify-out/graph.json` must never fail a request. Rillan logs the degradation and continues with chunk/vector retrieval.
   - Graphify presence may enrich retrieval results but must never *replace* the chunk/vector path as the primary signal.

2. **Confidence tags are respected.**
   - Graphify edges carry one of three confidence tags: `EXTRACTED`, `INFERRED`, `AMBIGUOUS`.
   - By default Rillan traverses only `EXTRACTED` edges. `INFERRED` and `AMBIGUOUS` edges are opt-in via explicit project configuration, and the confidence tag used is recorded in the retrieval debug metadata so a request can be explained.
   - A Rillan consumer must never silently treat `INFERRED` or `AMBIGUOUS` edges as ground truth.

3. **Storage boundary is preserved.**
   - Rillan reads graphify artifacts in-place from the project's gitignored `graphify-out/` directory.
   - Rillan does **not** copy nodes, edges, or derived features into its own SQLite store or any persistent state. Any cached projection of graphify data is explicitly scoped to a single request's lifecycle or, if we later decide otherwise, documented as ephemeral operational cache.
   - This preserves the invariant that the index root (ADR-003) is the single authoritative local mirror of repository content.

4. **Freshness is detected, never masked.**
   - Before each use, Rillan checks the sentinel conventions graphify already maintains — notably `graphify-out/.needs_update` and the mtime of `graph.json` relative to the configured index root.
   - If the graph is missing or stale, Rillan falls back to chunk/vector retrieval for that request and logs a single-line warning keyed by the staleness reason. Use of a stale graph is permitted only when the operator has explicitly opted in, and in that case the staleness must appear in the retrieval debug metadata so downstream audit (ADR-010) can correlate decisions made on stale structure.
   - Rillan does not itself rebuild graphify. Graph rebuilding stays with graphify's own git hooks and its `/graphify --update` command.

5. **Local-first fit is non-negotiable.**
   - Graphify integration must not introduce network calls, background watchers, or daemons that outlive request scope. File reads are synchronous and request-scoped.
   - Graphify must not become a precondition for `rillan serve`, `rillan init`, or `rillan index`. The daemon starts, indexes, and serves requests with or without a graph present.
   - The integration must not widen the local-bind surface (ADR-002) or relax the manual-indexing contract (ADR-003).

6. **Integration seams are explicit.**
   - Graph consumption happens inside `internal/retrieval/` (or a sibling `internal/graphify/` package fronting it), after chunk/vector retrieval produces candidate chunks and before the compiled context is handed to request preparation.
   - The consumer is a small adapter that takes the candidate set, optionally expands it along `EXTRACTED` edges (bounded in hop count and total additions), and annotates the compiled context with the structural provenance. It has no knowledge of specific providers, policy, or the HTTP layer.
   - The adapter is a seam for testing. Concrete traversal strategies (neighborhood expansion, ADR↔code linking, symbol-level enrichment) are introduced by the follow-on milestone and must stay local to this adapter.

7. **Audit and policy stay authoritative.**
   - Retrieval decisions influenced by graphify are tagged in the existing retrieval debug metadata so audit (ADR-010) and policy tracing (ADR-012) can see what was expanded and why.
   - Policy continues to be able to force local-only routing or redact content regardless of what graphify's edges say; graphify cannot override policy verdicts.

## Consequences

- **Retrieval fidelity.** When graphify is present and fresh, structural expansion of candidate chunks gives the compiled context better recall for symbol-centric questions and for queries that need to cross file boundaries along call or reference edges. ADR↔code semantic edges in particular are difficult to surface from chunk/vector retrieval alone.

- **Staleness risk.** Because Rillan does not rebuild the graph, there is always a window where the graph lags behind the code — between a commit and the post-commit hook completing, or between an ADR edit and a manual `/graphify --update`. The decision's freshness rules make staleness visible in audit and debug traces, but consumers should still assume the graph can be mildly out of date even in the "fresh" state.

- **Hook coupling.** The integration inherits graphify's git-hook rebuild model. If a user disables or misconfigures graphify's hooks, Rillan's structural enrichment silently degrades to the fallback path. This is acceptable under the optionality contract, but it means the user can't infer graph freshness from Rillan alone — they must check graphify's own status.

- **Operational simplicity.** By refusing to own graphify's lifecycle, Rillan stays narrow. The daemon has no new background goroutine, no new config for index scheduling, no new authoritative state. The cost is that graphify is a parallel system the user has to understand; the benefit is that neither tool has to absorb the other.

- **Interaction with existing ADRs.**
  - **ADR-002** (localhost bind). Unchanged — graphify reads are local file I/O; no new network surface.
  - **ADR-003** (one explicit root, manual indexing, SQLite as authoritative local mirror). Preserved by the storage boundary rule — graphify data never enters Rillan's SQLite store.
  - **ADR-010** (append-only audit JSONL). Extended via metadata tagging only; no new audit sinks.
  - **ADR-012** (tokenizer-backed outbound minimization). Compatible — structural expansion runs before minimization, and the same policy trace surfaces both. Minimization may eventually use graphify's structural signals to decide which symbols are load-bearing, but that is a follow-on decision, not part of this ADR.

- **Follow-on work.** A dedicated milestone (working name `M-GRAPH`) will implement the adapter, project-config surface for opt-in edge confidence tiers, retrieval-metadata tagging, and tests that exercise both the present-and-fresh and absent-or-stale paths. This ADR is the precondition for that milestone, not its design document.
