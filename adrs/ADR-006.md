# ADR-006: Multi-Provider Host with Preset-Based Capability Declaration

## Status

Accepted

## Context

Rillan needs to support multiple upstream LLM providers concurrently — a user may have OpenAI for general work, a local Ollama instance for sensitive code, and an xAI endpoint for experimentation. Each provider has different capabilities, authentication methods, and API quirks, but the inbound API surface is always OpenAI-compatible `/v1/chat/completions`.

A naive approach would create a separate adapter type for each upstream service. This leads to a combinatorial explosion as new providers are added. Meanwhile, most cloud providers already speak OpenAI-compatible HTTP, differing only in endpoint URL, auth header, and model names.

## Decision

Rillan uses a **provider host** (`providers.Host`) that holds a map of named providers, each implementing a single `Provider` interface with three methods: `Name()`, `Ready()`, and `ChatCompletions()`. The host is constructed at daemon startup from the resolved runtime config and serves as the single point of provider access.

Providers are grouped into **families** that share a single adapter implementation:

- `openai_compatible` — a shared HTTP client covering OpenAI, xAI, DeepSeek, Kimi, z.ai, and any other service that accepts OpenAI-shaped requests. One adapter, many endpoints.
- `anthropic` — a native adapter that translates from OpenAI request shape to Anthropic's `/v1/messages` API.
- `ollama` — a local adapter wrapping Ollama's HTTP inference API.

To reduce configuration boilerplate, Rillan ships **bundled presets** for known providers. Each preset defines the canonical endpoint, auth strategy, default model, model pins, and capability tags for a provider. When a user runs `rillan llm add work-gpt --preset openai`, the preset fills in defaults; only the provider name and API key are required.

Each provider declares **capabilities** (`chat`, `reasoning`, `tool_calling`) either explicitly in config or inherited from its preset. These capability tags are consumed by the routing engine to gate eligibility — a request requiring `tool_calling` will not be routed to a provider that doesn't advertise it.

## Consequences

- Adding a new OpenAI-compatible provider (e.g., a new cloud vendor) requires only a new preset definition — no new adapter code.
- The `Provider` interface is intentionally narrow (three methods). This keeps adapter implementations small and testable.
- Capability tags are declared, not discovered. If a provider adds new capabilities, the user must update their config or the bundled preset must be updated. This is a conscious trade-off: static declaration is predictable and auditable; runtime capability probing would add latency and non-determinism.
- The host validates at startup that the default provider exists and all declared providers can be constructed, failing fast on misconfiguration.
- Project-level `llm_allowed` lists filter the host's providers, letting repos restrict which providers receive their code.
