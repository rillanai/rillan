# ADR-007: Three-Tier Configuration with Validation Modes

## Status

Accepted

## Context

Rillan serves multiple use cases from a single binary: running the HTTP daemon (`serve`), rebuilding the local index (`index`), and inspecting status (`status`). Each command needs different config completeness — `serve` requires valid provider credentials, `index` only needs an index root path, and `status` can work with minimal config. Forcing full validation for every command would block users from indexing a codebase before they've configured a provider.

Separately, configuration needs to work across three scopes: daemon-wide settings, per-repository behavior, and machine-level policy enforcement. These scopes have different owners (the user, the project, and the organization) and different trust levels.

## Decision

Rillan implements a three-tier configuration hierarchy, each tier loaded independently with its own schema, defaults, and validation:

1. **Global config** (`config.yaml`) — daemon settings, named LLM providers, named MCP endpoints, index settings, retrieval defaults, local model configuration. Written by `rillan init`, mutated by CLI commands. Loaded with `config.Load()`.

2. **Project config** (`.rillan/project.yaml`) — repo-local classification, provider allowlists, skill activation, routing preferences, and custom instructions. Loaded with `config.LoadProject()`.

3. **System config** (`system.yaml`) — machine-level policy rules and identity patterns. Encrypted at rest using a keyring-derived key. Loaded with `config.LoadSystem()`, which rejects plaintext policy data and decrypts the payload before parsing.

The loading sequence for the global config follows a deterministic override order: hardcoded defaults → YAML file → environment variable overrides. Environment variables always win, supporting CI/automation use cases without requiring config file mutation.

Each CLI command specifies a **validation mode** (`ValidationModeServe`, `ValidationModeIndex`, `ValidationModeStatus`) that controls which fields are required:

- **Serve** — full validation including provider credentials, endpoint URLs, and transport configuration.
- **Index** — validates only that `index.root` is set.
- **Status** — no additional validation beyond basic schema sanity.

Common fields (server host/port, log level, index settings, retrieval bounds) are validated in all modes.

## Consequences

- Users can `rillan index` immediately after `rillan init` without configuring a provider, lowering the barrier to trying retrieval.
- Environment variable overrides make Rillan usable in containers and CI without writing config files, while keeping the YAML file as the primary human-readable configuration.
- The three tiers have different trust levels: global config is user-writable, project config is repo-scoped (potentially committed), and system config is encrypted. This prevents project configs from escalating privilege or overriding machine policy.
- Adding new config fields requires deciding which validation modes enforce them. The default is "validate in serve mode only" unless the field affects indexing or status.
- Project config can restrict the global provider list via `llm_allowed` but cannot add providers that don't exist in the global config, maintaining the global config as the source of truth for provider registration.
