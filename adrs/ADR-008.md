# ADR-008: Policy Evaluation with Four-Verdict Model

## Status

Accepted

## Context

Rillan proxies user requests to external LLM providers. Requests may contain API keys, private keys, PII, or trade-secret code that should not leave the local machine. The daemon needs a policy layer that can detect sensitive content, decide what to do about it, and enforce that decision before the request reaches a provider.

Policy evaluation must be fast (inline with every request), deterministic (same input always produces the same verdict), and hierarchical (scan results can override project settings, and system policy can override both).

## Decision

The policy system uses a **four-verdict model** with two evaluation phases:

**Verdicts** (ordered by severity):

- `Allow` — request proceeds unmodified.
- `Redact` — sensitive content is replaced with safe placeholders before forwarding.
- `LocalOnly` — request may only be routed to local providers (e.g., Ollama). Remote providers are excluded.
- `Block` — request is rejected entirely. No provider receives it.

**Evaluation phases:**

- **Preflight** — runs before routing. Determines the verdict, applies redaction to the request body, and optionally constrains retrieval (e.g., reducing `top_k` for remote-bound requests).
- **Egress** — runs after provider selection, before the outbound HTTP call. Applies the same verdict logic as a final gate.

**Decision hierarchy** (first match wins):

1. **Scan results** — a regex-based scanner (`policy.Scanner`) runs against the serialized request body. Each rule maps to either `FindingActionRedact` or `FindingActionBlock`. If any finding triggers `Block`, the verdict is `Block`. If findings exist but none block, the verdict is `Redact` and matched spans are replaced with safe placeholders.
2. **Classifier sensitivity** — if intent classification is available and reports `trade_secret` sensitivity, the verdict is `LocalOnly`.
3. **Project classification** — if the project config declares `classification: trade_secret`, the verdict is `LocalOnly`.
4. **System policy** — system config can set `force_local_for_trade_secret`, enforcing `LocalOnly` regardless of other signals.

The scanner uses a **non-overlapping match selection** algorithm: all rule matches are sorted by start position (stable, longer matches preferred), then overlapping matches are discarded. Redaction is applied in reverse order to preserve string offsets.

Default scan rules cover OpenAI API keys, GitHub tokens, bearer tokens, AWS access keys, and private key blocks. Rules are composable — custom rules can be added via the scanner constructor.

Every evaluation produces a `PolicyTrace` recording the phase and which decision source determined the verdict.

## Consequences

- Four verdicts provide a clear escalation path. Code that checks `verdict == VerdictAllow` knows exactly what "allow" means; there are no ambiguous intermediate states.
- Regex-based scanning avoids ML dependencies and runs in microseconds. The trade-off is that patterns must be maintained manually and may produce false positives on content that looks like a key but isn't.
- The hierarchy ensures scan results (concrete evidence of secrets) always override project metadata (which could be misconfigured). System policy can further tighten but never loosen verdicts.
- Redaction modifies the request body in-place, so the provider receives a sanitized copy. The original body is preserved in the evaluation input for audit purposes.
- The routing engine consumes policy verdicts as hard gates — a `Block` verdict means no candidate is eligible, and `LocalOnly` filters out all remote candidates before scoring begins.
