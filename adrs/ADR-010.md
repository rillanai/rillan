# ADR-010: Audit Ledger as Append-Only JSONL

## Status

Accepted

## Context

Rillan proxies requests to external LLM providers and enforces policy decisions that may redact, block, or reroute content. Operators and developers need to understand what happened to a request after the fact — which provider was selected, what the policy verdict was, whether content was redacted, and whether the upstream call succeeded or failed.

The audit system needs to be simple enough that it doesn't become a maintenance burden for a daemon that runs on developer laptops, but complete enough to answer "what did Rillan do with my request?" after the fact.

## Decision

Rillan writes audit events to an **append-only JSONL file** (`$DATA_DIR/audit/ledger.jsonl`). Each line is one self-contained JSON object representing a single event.

**Event types:**

- `remote_egress` — a request was sent to an upstream provider.
- `remote_deny` — a request was blocked by policy before reaching a provider.
- `agent_action_proposed` — the agent proposed an action for approval.
- `agent_action_approved` — a proposed action was approved.
- `agent_action_denied` — a proposed action was denied.

**Event fields** include: timestamp (UTC), type, request ID, provider, model, verdict, reason, route source, and optional error. Request and response bodies are **not stored** — instead, SHA256 hashes of the outbound request body and inbound response body are recorded for correlation and tamper evidence.

All writes are serialized through a **mutex** on the audit store. The file is opened in append mode for each write and closed immediately after, so the ledger remains valid even if the daemon crashes between writes.

The file is created with `0600` permissions (owner-only read/write).

## Consequences

- JSONL is human-readable, greppable, and streamable. Developers can `tail -f` the ledger or pipe it through `jq` without special tooling.
- Append-only semantics mean events are never modified or deleted by the daemon. This provides a reliable timeline even if the daemon restarts.
- SHA256 hashes provide body correlation (matching a request to its response, or comparing what was sent vs. what was logged elsewhere) without storing potentially sensitive request content in plaintext on disk.
- The mutex serializes all writes, which is acceptable for a local daemon (tens of requests per minute, not thousands per second). High-throughput use cases would need a different design.
- The ledger grows unbounded. Retention, rotation, and compression are not built in and would need to be handled externally (e.g., logrotate) or added as a future feature.
- The `Recorder` interface allows alternative audit backends to be swapped in (e.g., a database or remote service) without changing the callers.
