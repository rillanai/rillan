# ADR-005: Deterministic Routing with Decision Tracing

## Status

Accepted

## Context

Rillan supports multiple named LLM providers simultaneously. Each request needs a single provider selected from the available candidates. Common approaches include round-robin, random selection, or load-based heuristics, but all of these make debugging difficult because the same request can land on different providers across runs.

Rillan also enforces policy verdicts and project-level routing preferences that constrain which providers are eligible. The routing decision needs to be explainable — operators and developers need to understand why a request went to a specific provider, especially when policy forces local-only handling or rejects a candidate for missing capabilities.

## Decision

Provider selection is implemented as a pure, stateless function (`routing.Decide`) that always returns the same result given the same inputs. The function does not read external state, use randomness, or cache results between calls.

The decision pipeline evaluates candidates in a fixed order:

1. **Policy eligibility** — candidates violating the active policy verdict (e.g., remote providers when verdict is `local_only`) are rejected.
2. **Capability gating** — candidates missing required capabilities (e.g., `tool_calling`, `multimodal`) are rejected.
3. **Route preference scoring** — candidates are scored based on the resolved preference (`prefer_local`, `prefer_cloud`, `local_only`, `auto`), with preferences resolved from task-type overrides, project defaults, or the built-in `auto` fallback in that order.
4. **Model affinity** — when the request specifies a model, candidates advertising that model via `model_pins` or `default_model` are preferred. If any eligible candidate matches, non-matching candidates are excluded entirely.
5. **Task-strength scoring** — candidates are scored by how well their declared capabilities match the classified action type (e.g., `tool_calling` scores higher for code generation tasks).
6. **Stable tie-break** — when candidates are otherwise equal, selection falls through to lexicographic comparison of provider ID.

Every decision produces a complete `DecisionTrace` recording which candidates were eligible, rejected, or selected, and why. The trace includes per-candidate details: eligibility, rejection reason, model match status, missing capabilities, preference score, and task strength.

## Consequences

- The same config and request always produce the same provider selection, making routing decisions reproducible across environments and debuggable from logs alone.
- The route decision itself is pure and stateless, but the current app builds the candidate catalog and status table at startup and reuses them per request. This keeps request-time routing deterministic while avoiding unnecessary per-request catalog reconstruction.
- The full decision trace enables audit logging and operator debugging without requiring request replay.
- Adding new scoring dimensions requires extending the `compareCandidateState` sort function, which must preserve the existing priority order.
- Route preferences are resolved hierarchically (task-type > project default > built-in `auto` fallback), giving projects fine-grained control over where different kinds of work land.
