# ADR-011: Local Intent Classification with Graceful Degradation

## Status

Accepted

## Context

Rillan's routing and policy systems benefit from knowing what kind of task a request represents — code diagnosis, code generation, refactoring, review, general Q&A, etc. This classification informs routing preferences (e.g., code generation prefers providers with `tool_calling` capability) and policy decisions (e.g., trade-secret sensitivity forces local-only routing).

Classification could be done by the upstream LLM provider, but that would require sending the full request content to a remote service before deciding whether the content should leave the machine — defeating the purpose of local-first policy enforcement. It could also be done with static heuristics, but natural language intent is too varied for pattern matching.

## Decision

Intent classification runs locally via a **small LLM** on the user's Ollama instance. The classifier sends the chat messages to a local model with a structured prompt requesting JSON output with five fields: `action_type`, `sensitivity`, `requires_context`, `execution_mode`, and `confidence`.

The classifier enforces a **minimum confidence threshold** (default 0.5). If the model's self-reported confidence is below this threshold, the classification is discarded with `ErrLowConfidence` rather than acting on a weak signal.

All classifiers are wrapped in a **`SafeClassifier`** decorator that catches any error — model unavailability, malformed JSON, low confidence, timeout — and returns `nil` instead of propagating the error. This makes classification a best-effort enhancement: if it works, routing and policy benefit from richer signal; if it fails, the request proceeds with default routing and classification-independent policy.

The classifier output feeds into two downstream systems:

- **Policy evaluation** — `sensitivity: trade_secret` triggers `VerdictLocalOnly`.
- **Routing** — `action_type` determines task-strength scoring (e.g., code generation weighs `tool_calling` capability higher) and can resolve to a task-specific route preference via project config.

## Consequences

- Classification never blocks request processing. The `SafeClassifier` pattern ensures that Ollama being down, the model returning garbage, or the classification being low-confidence all result in graceful degradation to default behavior.
- Running classification locally means request content never leaves the machine for the purpose of classification, preserving the local-first security model.
- The confidence threshold prevents acting on weak classifications. A model that is uncertain about sensitivity will not accidentally force local-only routing.
- The classification prompt is hardcoded. Customizing it (e.g., adding project-specific action types) requires changing the source. This is a deliberate simplicity trade-off for now — the prompt covers the common taxonomy and the `SafeClassifier` pattern means a bad prompt cannot break the system.
- Classification accuracy depends on the quality of the local model. Small models (e.g., `qwen3:0.6b`) may produce lower-quality classifications than larger ones, but the confidence threshold and graceful degradation mitigate this.
